## 第一步：D0 紧急响应确认（24小时内必须完成）

**问：这个问题需要紧急响应吗？**

### 触发条件（满足任一即需紧急响应）

| 条件 | 说明 |
|------|------|
| 安全相关 | 影响驾驶安全、功能安全 |
| 客户产线停线 | 已造成客户产线停止 |
| 法规违规 | 涉及法规要求不满足 |
| 客户明确要求 | OEM 发出的 8D 请求函 |

### D0 操作：症状收集

1. 记录**最先发现问题的地点**（客户现场 / 来料检验 / 内部过程）
2. 记录**受影响的产品/零件号**（Part Number + Lot/Batch 号）
3. 记录**数量和比例**（不良数量 / 总数量 = PPM 或 %）
4. 立即识别是否需要 **Emergency Response Action (ERA)**

> **ERA 原则**：在正式 8D 完成之前，先保护客户不要收到更多坏件。

## 第二步：D1 组建团队（1天内）

**问：谁来负责解决这个问题？**

### 核心角色

| 角色 | 职责 | 汽车电子典型人选 |
|------|------|----------------|
| **Team Leader** | 熟悉 8D 流程，引导团队，不是最懂技术的人 | 质量工程师 / 项目经理 |
| **Champion/Sponsor** | 能拍板调配资源，审批变更 | 部门经理 / 工程主管 |
| **Core Team** | 3人：产品 + 工艺 + 数据 | 工艺工程师 + 质量 + 生产主管 |
| **SME（按需）** | 主题专家，支持鱼骨图分析 | 设备供应商 / 设计工程师 |

### D1 产出
- [ ] 团队成员名单（含职责分工）
- [ ] Champion 确认（签字或邮件记录）
- [ ] 团队会议时间确定

## 第三步：D2 问题描述（5W2H + Is/Is Not）

**问：到底发生了什么？**  
**这是 8D 中最关键的一步——描述不清楚，后续全错。**

### 3.1 5W2H 描述法

| 问题 | 填写内容 | 示例 |
|------|---------|------|
| **What**（是什么） | 具体零件 + 缺陷类型 | TBOX 外壳塑料件开裂 |
| **When**（何时） | 发现日期 + 班次 | 2024-03-15，白班 08:30 |
| **Where**（何处） | 发现工位 / 客户工厂 | 客户装配线工位 #3 |
| **Who**（谁发现） | 检验员 / 客户 | 客户质量工程师李工 |
| **Why**（趋势） | 偶发还是频发 | 首次报告 / 批量性 |
| **How**（如何确认） | 检测方法 + 数据 | 外观检验 + 3D CT 扫描 |
| **How many**（多少） | 数量 + 比例 | 5/500 = 1% = 10000 PPM |

### 3.2 Is / Is Not 分析（关键！）

| 维度 | Is（符合问题的） | Is Not（排除的） |
|------|-----------------|-----------------|
| **产品** | PN-ABC-123 Rev.C | PN-ABC-123 Rev.B（历史批次） |
| **工位** | 装配工位 #3 | 装配工位 #1, #2 |
| **班次** | 白班（07:00-15:00） | 夜班、中班 |
| **供应商** | 我司（内制件） | 外协供应商 |
| **时间范围** | 2024-W11 起 | W10 之前的批次 |
| **现象** | 开裂 + 尺寸超差 | 单纯外观刮花 |

> **原则**："Is Not" 帮助排除可能性，缩小根因搜索范围。  
> **进阶**：每填一行"Is Not"，追问"为什么这个可以排除"，用数据或事实说话。

### D2 产出
- [ ] 5W2H 完整填写
- [ ] Is/Is Not 表格（至少 6 个维度）
- [ ] 问题陈述（1 句话概括，如："2024-03-15 起，客户装配线发现 TBOX 外壳开裂，5/500 = 1%，影响白班装配"）

## 第四步：D3 临时遏制措施（24小时内必须开始）

**问：怎么确保客户不再收到更多坏件？**

### 遏制措施清单

| 措施 | 具体动作 | 负责人 | 完成时间 |
|------|---------|-------|---------|
| 隔离 | 将嫌疑批次零件从仓库/产线移出，贴红色标签 | 仓库 | 即日 |
| 100% 检验 | 对嫌疑批次全数检验，好件放行，坏件报废 | QC | 2天内 |
| 客户库存排查 | 联系客户，确认是否用到嫌疑零件，提供追溯清单 | 业务/质量 | 3天内 |
| 新产批次控制 | 改善后新批次必须首批次 100% 检验才能出货 | 生产 | 即日 |

### ⚠️ 常见错误

> **错误**：只在我司产线上改善，没有追溯客户端和运输中库存。  
> **正确**：遏制必须覆盖整个供应链——从现在往前查。

### D3 产出
- [ ] 嫌疑批次隔离确认（含数量和位置）
- [ ] 100% 检验执行记录
- [ ] 客户库存追溯结果
- [ ] Clean Point（从哪一批起可以放心出货）记录

## 第五步：D4 根因分析（核心步骤）

**问：为什么这个问题会发生？为什么没有早点发现？**

### 5.1 三层根因（必须全部分析）

| 层次 | 问题 | 分析工具 |
|------|------|---------|
| **技术根因** | 这颗零件为什么坏？ | 5-Why、鱼骨图（6M） |
| **质量保证根因** | 为什么没有检测出来？ | 过程审核、检测能力分析 |
| **PDP/系统根因** | 为什么设计/开发阶段没有预防？ | FMEA回顾、DFMEA检查 |

### 5.2 5-Why 分析法（示例）

```
问题：TBOX 外壳开裂

Why1：为什么外壳开裂？
→ 塑料注塑时冷却不均匀，内部产生应力集中

Why2：为什么冷却不均匀？
→ 模具冷却水道堵塞，水流量降低

Why3：为什么水道堵塞？
→ 冷却水没有过滤，杂质堆积

Why4：为什么没有过滤？
→ 设备采购时未要求水路过滤装置

Why5：为什么未要求？
→ 设备技术协议中未明确冷却水质要求

技术根因：模具冷却水道无过滤装置
质量保证根因：设备验收时未将水路清洁度纳入检测项
```

### 5.3 鱼骨图（6M 分析法）

```
                人     机     料     法     环     测
                ↓      ↓      ↓      ↓      ↓      ↓
            操作员   注塑机   塑料粒子  工艺参数  车间温湿度  三坐标
            培训    模具状态  批次差异  压力曲线  洁净度    测量系统
```

### 5.4 根因验证（关键！）

> **根因必须可"开关验证"**：打开原因，问题复现；关闭原因，问题消失。  
> 如果无法验证，说明根因假设还不成立，继续分析。

### 5.5 Escape Point（逃脱点）分析

**问：这个问题在哪个环节本应被发现但漏掉了？**

- 最接近客户的那个检测点 → 为什么没有拦住？
- 更新 Control Plan（控制计划）时重点标注

### D4 产出
- [ ] 技术根因（文字 + 5-Why 链）
- [ ] 质量保证根因
- [ ] PDP/系统根因（如适用）
- [ ] Escape Point 记录
- [ ] 根因验证数据

## 第六步：D5 永久改善措施（PCA）

**问：怎么从根本上解决这个问题？**

### 选择原则

| 原则 | 说明 |
|------|------|
| **消除** | 完全移除根本原因（最佳） |
| **替代** | 用更安全的材料/工艺替换 |
| **控制** | 增加检测手段拦截（次优） |

### FMEA 评估

对每个候选方案，用 FMEA 评估：
- **Severity**（严重度）：问题还会发生吗？
- **Occurrence**（发生频度）：新方案可靠性如何？
- **Detection**（探测度）：新检测手段能拦住吗？

RPN = S × O × D，目标是 RPN 显著降低。

### D5 产出
- [ ] 选定 PCA 方案（含替代方案对比）
- [ ] FMEA 评估记录
- [ ] 风险评估结论

---

## 第七步：D6 实施与验证

**问：改善措施真的有效吗？**

### 验证方法

| 验证方式 | 适用场景 | 合格标准 |
|---------|---------|---------|
| 首批 100% 检验 | 新批次量产前 | 0 不良 |
| 试生产 | 改善工艺参数后 | 不良率 < 目标 |
| 客户端装机验证 | 安全件、关键件 | 规定里程/时间无异常 |
| 数据监控 | PPM 趋势跟踪 | 连续 3 个月达标 |

### 实施时间线（参考 MAHLE 1-5-20 规则）

| 时间 | 要求 |
|------|------|
| **D+1 工作日** | D1-D3 完成（团队 + 问题描述 + 遏制） |
| **D+5 工作日** | D1-D5 完成（根因 + PCA 方案确定） |
| **D+20 工作日** | D1-D8 完成（改善实施 + 验证 + 预防） |

### D6 产出
- [ ] 改善实施计划（Gantt 图或里程碑表）
- [ ] 验证数据（改善前后对比）
- [ ] 客户沟通记录

## 第八步：D7 预防再发

**问：这个问题以后怎么预防？**

### 预防措施清单

| 领域 | 具体动作 | 负责人 |
|------|---------|-------|
| **FMEA 更新** | 更新 DFMEA/PFMEA，将根因加入 | 工艺工程师 |
| **控制计划更新** | 在 Escape Point 增加检测项 | 质量工程师 |
| **SOP 更新** | 修订作业指导书，纳入新工艺要求 | 生产主管 |
| **培训** | 对相关岗位重新培训，记录签到 | 培训专员 |
| **横向展开（Yokoten）** | 检查其他产品线是否有类似问题 | 质量工程师 |

### D7 产出
- [ ] FMEA 更新记录（版本号 + 变更内容）
- [ ] 控制计划更新确认
- [ ] SOP 修订记录
- [ ] 横向展开检查结果
- [ ] 团队培训记录

---

## 第九步：D8 关闭与表彰

**问：这个问题真的结束了吗？**

### 关闭条件

- [ ] 所有 D1-D7 产出完成并归档
- [ ] 改善效果经数据验证（连续 3 个月 PPM 达标）
- [ ] 客户书面关闭（或在客户系统确认关闭）
- [ ] Lessons Learned 记录完成

### Lessons Learned 模板

```markdown
## Lessons Learned — [问题简述]

**根本原因**：[一句话]
**主要改善措施**：[一句话]
**如果重来会怎么做的 3 件事**：
1. ...
2. ...
3. ...
**对其他产品线的建议**：...
```

### D8 产出
- [ ] 8D 报告归档（含客户关闭确认）
- [ ] Lessons Learned 文档
- [ ] Before/After 对比数据
- [ ] 团队表彰记录

## 快速参考卡（D0-D8 总览）

```
D0 紧急响应  → 有没有安全/停线/法规问题？→ 立即启动 ERA

D2 问题描述  → 5W2H + Is/Is Not（6维度）

D4 根因分析  → 技术根因 + QA根因 + PDP根因 + Escape Point

D7 预防再发  → FMEA更新 + SOP更新 + 横向展开
D8 关闭表彰  → 归档 + Lessons Learned + 团队表彰
```

## 常见错误清单

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| D2 描述模糊 | 根因找偏 | 5W2H 每项必须填，数据说话 |
| D3 只顾自己 | 客户继续收坏件 | 覆盖整个供应链追溯 |
| D4 只分析一层根因 | 表面解决，复发 | 三层根因（技术+QA+PDP）全部过一遍 |
| D5 只做控制不做消除 | 根因仍在，只是加了检测 | 优先消除，其次替代，最后控制 |
| D7 不更新 FMEA/SOP | 新人不知道，新批次重复犯错 | 系统文件必须同步更新 |
| D8 不收集 Lessons Learned | 组织记忆丢失 | 归档标准动作，不能省 |
